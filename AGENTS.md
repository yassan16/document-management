# リポジトリガイドライン

## 目的

このリポジトリは、ソフトウェアエンジニアリングに関する日本語のナレッジベースです。主な変更対象は、Markdownドキュメント、図表、参考リンクです。

リポジトリの分類体系は `README.md` を確認してください。編集対象に最も近いカテゴリの `README.md` を、その領域の索引として使用してください。

## リポジトリ構成

- `01-AI/`: AIツールと技術
- `02-フレームワーク/`: アプリケーションフレームワーク
- `03-設計/`: アーキテクチャ、モデリング、UML、UI/UX
- `04-開発手法/`: 開発プロセスとワークフロー
- `05-インフラ/`: インフラ、ネットワーク、開発ツール
- `06-DevOps/`: CI/CD、IaC、可観測性、GitOps、および関連する手法
- `99-一時フォルダ/`: 一時的な計画資料。Gitの管理対象外

既存の番号付きトップレベルディレクトリ名を維持してください。タスクで明示されていない限り、カテゴリを再編成しないでください。

## 分類方針

- AIツールやAI活用技術に関するトピックは `01-AI/` に配置してください。
- フレームワーク固有の動作は `02-フレームワーク/` に配置してください。
- 成果物の形や構造に関するトピックは `03-設計/` に配置してください。
- 開発作業の進め方に関するトピックは `04-開発手法/` に配置してください。
- ランタイム、クラウド、ネットワーク、コンテナ、ツールに関するトピックは、既存の周辺ドキュメントに合わせて `05-インフラ/` または `06-DevOps/` に配置してください。
- 分類が不明確な場合は、新しいディレクトリを作成する前に、ルートと各カテゴリの `README.md` を確認してください。

## ドキュメント作成ルール

- 周辺のセクションが意図的に英語で書かれている場合を除き、ドキュメントは日本語で記述してください。
- 周辺ファイルの用語、見出しの形式、ファイル名の規則を維持してください。
- 独立した主要ドキュメントには、次の内容を順番に含めてください。
  1. 1つの `#` タイトル
  2. `## 概要` セクション
  3. 正しく機能するアンカーリンクを含む `## 目次`
  4. 論理的な階層に整理された詳細セクション
  5. 情報源へのリンクを含む `## 参考` または `## 参考資料` セクション
- その概念が何であるかに加えて、なぜ重要なのかも説明してください。
- 理解の助けになる場合は、具体例、コマンド、コードスニペット、表、図を含めてください。
- 公式ドキュメント、仕様書、標準規格、原著論文などの一次情報を優先してください。時間の経過で変化する可能性がある内容は、記述前に確認してください。
- ローカルファイルへのリンクには、リポジトリ内の相対リンクを使用してください。ファイルを改名または移動した場合は、そのファイルを参照するすべてのリンクと、関連するディレクトリ構成例を更新してください。
- ドキュメントを追加、削除、改名、または大幅に変更した場合は、最も近いカテゴリの `README.md` を更新してください。トップレベルの分類体系や概要が変わる場合は、ルートの `README.md` も更新してください。
- タスクで必要とされていない限り、自動生成ファイルやバイナリファイルを編集しないでください。

## Mermaid

用途に応じて図の種類を使い分けてください。

- `erDiagram`: エンティティ、テーブル、カラム、リレーション
- `sequenceDiagram`: API呼び出しや時系列の処理
- `flowchart`: 判定、状態遷移、コンポーネント間の関係
- Markdownテーブル: 簡潔な比較、フィールドやステータスの一覧

Mermaidのノード内で改行する場合は `<br/>` を使用してください。描画に失敗する可能性があるため、`\n` は使用しないでください。ノードのラベルは簡潔にし、IDが重複しないようにしてください。

## 検証

リポジトリ全体に共通するビルドコマンドやテストコマンドはありません。変更した範囲に応じて、次の項目を検証してください。

1. `git diff --check` と `git diff` の結果を確認する。
2. 新規または変更したローカルのMarkdownリンクが正しく参照できることを確認する。
3. 目次と見出しが一致していることを確認する。
4. Mermaidブロックの構文と、改行に `<br/>` が使用されていることを確認する。
5. `git status --short` を実行し、無関係なファイルが変更されていないことを確認する。

対応する確認を実際に行っていない場合は、リンクや図を検証済みと報告しないでください。

## 変更範囲

- 変更は依頼されたトピックに限定してください。
- ワーキングツリーに存在するユーザー作成の変更を維持してください。
- 明示的に依頼されていない限り、広範囲な文章修正、一括改名、カテゴリ再編成を行わないでください。
- シークレット、認証情報、個人トークン、特定のマシンに依存するパスを、Git管理対象のドキュメントに追加しないでください。

## 1. SSRとは何か

SSRとは、**Server Side Rendering** の略です。

ユーザーがページにアクセスしたタイミングで、サーバー側がHTMLを生成してブラウザに返す仕組みです。

簡単に言うと、SSRは以下のような方式です。

```txt
ユーザーがページにアクセス
↓
サーバー側でHTMLを生成
↓
ブラウザにHTMLを返す
↓
画面が表示される
```

通常の静的サイトのように、あらかじめ作成済みのHTMLを返すのではなく、アクセス時にサーバー側で処理を行う点が特徴です。

---

## 2. SSG・CSR・SSRの違い

SSRを理解するには、SSGやCSRとの違いを見ると分かりやすいです。

| 方式  | HTMLを作る場所 | HTMLを作るタイミング | 主な用途                         |
| --- | --------- | ------------ | ---------------------------- |
| SSG | ビルド時      | デプロイ前        | ブログ、固定ページ、LP                 |
| CSR | ブラウザ      | アクセス後        | 管理画面、SPA、ログイン後画面             |
| SSR | サーバー      | アクセス時        | 動的ページ、SEO重視ページ、ログイン判定が必要なページ |

### SSG

SSGは、事前にHTMLを生成しておく方式です。

```txt
ビルド時にHTMLを作成
↓
ホスティングサービスに配置
↓
ユーザーアクセス時に作成済みHTMLを返す
```

固定ページや記事ページに向いています。

### CSR

CSRは、ブラウザ側のJavaScriptで画面を作る方式です。

```txt
ブラウザがHTMLとJavaScriptを読み込む
↓
JavaScriptがAPIからデータ取得
↓
ブラウザ上で画面を描画
```

管理画面やログイン後のアプリ画面に向いています。

### SSR

SSRは、アクセス時にサーバー側でHTMLを作る方式です。

```txt
ユーザーがアクセス
↓
サーバーがデータ取得やログイン判定を行う
↓
HTMLを生成
↓
ブラウザに返す
```

動的データやログイン状態に応じた表示が必要なページに向いています。

---

## 3. SSRの基本的な処理の流れ

SSRでは、ページアクセス時に以下のような流れで処理されます。

```txt
ユーザーがページにアクセス
↓
ブラウザがサーバーにリクエストを送る
↓
サーバーがリクエスト内容を確認する
↓
必要に応じてCookie、DB、APIなどを確認する
↓
サーバー側でHTMLを生成する
↓
生成したHTMLをブラウザに返す
↓
画面が表示される
```

ポイントは、**ブラウザにHTMLを返す前に、サーバー側で処理できる**ことです。

---

## 4. SSRを利用するメリット

SSRを使う主なメリットは以下です。

```txt
1. SEOに強い
2. 初回表示が速く感じやすい
3. ログイン状態やCookieを使った表示分岐がしやすい
4. 最新データを反映しやすい
5. APIキーやサーバー側処理を隠しやすい
```

---

## 5. メリット1：SEOに強い

SSRでは、サーバー側でHTMLを生成してからブラウザに返します。

そのため、検索エンジンがページを見たときに、最初からHTML内にコンテンツが含まれています。

```html
<h1>商品一覧</h1>
<p>このページではおすすめ商品を紹介しています。</p>
```

これにより、検索エンジンがページ内容を理解しやすくなります。

特に、以下のようなページではSEO上のメリットがあります。

```txt
商品一覧ページ
記事一覧ページ
カテゴリページ
検索結果ページ
レビュー一覧ページ
```

---

## 6. メリット2：初回表示が速く感じやすい

CSRでは、最初にHTMLとJavaScriptを読み込んでから、ブラウザ上で画面を作ります。

```txt
HTMLを読み込む
↓
JavaScriptを読み込む
↓
APIからデータ取得
↓
ブラウザで画面を描画
```

一方、SSRではサーバー側でHTMLを作ってから返します。

```txt
サーバー側でHTMLを生成
↓
ブラウザに返す
↓
画面が表示される
```

そのため、ユーザーから見ると、最初から内容のある画面が表示されやすくなります。

---

## 7. メリット3：ログイン状態やCookieを使った表示分岐がしやすい

SSRでは、画面を返す前にサーバー側でログイン状態を確認できます。

たとえば、ユーザーがマイページにアクセスした場合です。

```txt
ユーザーが /mypage にアクセス
↓
サーバーがCookieを確認
↓
ログイン済みか判定
↓
ログイン済みならマイページを返す
↓
未ログインならログインページへリダイレクト
```

つまり、SSRでは**ユーザーに画面を返す前に、サーバー側で「この人はログイン済みか？」を判断できる**ということです。

---

## 8. Cookieとは何か

Cookieとは、ブラウザに保存される小さな情報です。

ログイン済みのWebサービスでは、Cookieを使ってユーザーのログイン状態を判断することがあります。

イメージとしては以下です。

```txt
ブラウザにCookieが保存されている
↓
ページアクセス時にCookieもサーバーへ送られる
↓
サーバーがCookieを見てログイン状態を確認する
```

たとえば、以下のような判断に使われます。

```txt
ログイン済みか
どのユーザーか
会員ランクは何か
管理者か一般ユーザーか
```

---

## 9. SSRでログイン判定する場合

SSRでは、ログイン状態を確認してからHTMLを返せます。

### ログイン済みの場合

```txt
ユーザーAが /mypage にアクセス
↓
サーバーがCookieを確認
↓
ユーザーAだと判断
↓
ユーザーAの情報を取得
↓
ユーザーA用のHTMLを生成
↓
ブラウザに返す
```

表示例：

```txt
ようこそ、Aさん
お気に入り一覧
購入履歴
アカウント設定
```

### 未ログインの場合

```txt
未ログインユーザーが /mypage にアクセス
↓
サーバーがCookieを確認
↓
ログイン情報がない
↓
ログインページへリダイレクト
```

表示例：

```txt
ログインしてください
```

## 10. CSRでもログイン画面へ遷移できるのでは？という疑問

CSRでも、JavaScriptを使ってログイン判定を行い、未ログインならログイン画面へ遷移させることはできます。

```txt
ユーザーが /mypage にアクセス
↓
HTMLとJavaScriptを読み込む
↓
JavaScriptがログイン状態をAPIに確認
↓
未ログインなら /login に遷移
↓
ログイン画面を表示
```

そのため、単に「未ログインならログイン画面を表示する」だけなら、CSRでも実現可能です。

ただし、SSRとは判定のタイミングが違います。

---

## 11. CSRとSSRのログイン判定タイミングの違い

CSRの場合は、画面を返したあとにJavaScriptで判定します。

```txt
ページを返す
↓
JavaScriptを読み込む
↓
ログイン判定する
↓
必要ならログイン画面へ遷移する
```

SSRの場合は、画面を返す前にサーバー側で判定します。

```txt
ログイン判定する
↓
必要ならログインページへリダイレクトする
↓
画面を返す
```

大きな違いは、**SSRは最初のレスポンスを返す前に判断できる**ことです。

---

## 12. SSRがログイン判定で有利な理由

SSRがログイン判定で有利な理由は以下です。

```txt
1. 画面を返す前に判定できる
2. チラつきが起きにくい
3. 不要なAPI通信を減らせる
4. 最初から正しい画面を返せる
5. サーバー側に制御を寄せやすい
```

### 1. 画面を返す前に判定できる

SSRでは、ユーザーにHTMLを返す前にログイン状態を確認できます。

そのため、未ログインユーザーにマイページ用のHTMLを返さずに済みます。

### 2. チラつきが起きにくい

CSRでは、ログイン判定が終わるまでローディング画面が表示されたり、一瞬だけ別の画面が見えたりすることがあります。

SSRでは、最初から正しい画面を返せるため、このようなチラつきが起きにくくなります。

### 3. 不要なAPI通信を減らせる

CSRでは、実装によっては未ログインなのにマイページ用APIを呼んでしまうことがあります。

```txt
マイページ表示
↓
ユーザー情報APIを呼ぶ
↓
購入履歴APIを呼ぶ
↓
未ログインと判定
↓
ログイン画面へ移動
```

SSRでは、最初にログイン判定できます。

```txt
マイページへアクセス
↓
サーバー側でログイン判定
↓
未ログイン
↓
マイページ用データは取得しない
↓
ログインページへリダイレクト
```

### 4. 最初から正しい画面を返せる

SSRでは、ログイン済みならマイページ、未ログインならログインページというように、最初から正しいレスポンスを返せます。

### 5. サーバー側に制御を寄せやすい

画面上の表示制御だけでなく、サーバー側でアクセス制御を行いやすくなります。

ただし、セキュリティ上本当に重要なのは、SSRかCSRかではなく、**API側やサーバー側で認可を行うこと**です。

## 13. SSRを推奨するパターン

SSRは、以下のようなパターンで利用が推奨されます。

```txt
ログイン状態で表示内容が変わるページ
ユーザーごとに内容が違うページ
アクセス時点の最新データが必要なページ
SEOが重要で、かつ内容が動的なページ
OGPを動的に変えたいページ
```

## 14. 具体例1：マイページ

マイページは、ユーザーごとに表示内容が異なります。

```txt

Bさん → Bさんの購入履歴
未ログインユーザー → ログインページ
```

このように、アクセスしているユーザーによって返す内容を変えたい場合、SSRが向いています。

```txt
↓
ログイン済みなら購入履歴を取得
↓
購入履歴ページを生成
↓
未ログインならログインページへリダイレクト
```

このようなページでは、SSRにより最初から正しい内容を返しやすくなります。

## 16. 具体例3：ランキングページ

ランキングページのように、アクセス時点の最新データを表示したい場合もSSRが向いています。

```txt

↓
サーバー側で最新ランキングを取得
↓
ランキングHTMLを生成
↓
ブラウザに返す
```

ただし、ランキングの更新頻度が低い場合は、静的生成やキャッシュでも対応できる場合があります。

検索結果ページは、検索条件によって表示内容が変わります。

```txt

/search?category=yyy

```

検索条件に応じてサーバー側で結果を取得し、HTMLを生成できます。

```txt
検索条件を受け取る
↓
サーバー側で検索処理を行う
↓
検索結果HTMLを生成
↓
ブラウザに返す
```

SEOを重視する検索結果ページやカテゴリページでは、SSRが有効な場合があります。

## 19. SSRを使わなくてもよいパターン

SSRは便利ですが、すべてのページで必要なわけではありません。

以下のようなページでは、SSGや静的配信で十分なことが多いです。

```txt
会社概要
利用規約
プライバシーポリシー
広告・免責事項
固定のLP
静的な記事ページ
お問い合わせページ
```

これらは内容が頻繁に変わらないため、アクセスごとにサーバー側でHTMLを生成する必要性が低いです。

## 20. SSRの注意点

SSRにはメリットがありますが、注意点もあります。

```txt
1. 静的サイトより構成が複雑になる
2. アクセスごとに処理が走る
3. サーバー負荷やコストが増える可能性がある
4. キャッシュ設計が必要になる
5. 実行環境がSSRに対応している必要がある
```

### 1. 構成が複雑になる

静的サイトであれば、HTML、CSS、JavaScriptを配置するだけで配信できます。

SSRでは、サーバー側で処理を実行するための環境が必要になります。

### 2. アクセスごとに処理が走る

SSRは、アクセスのたびにHTMLを生成します。

そのため、静的配信よりもサーバー側の処理が増えます。

### 3. コストが増える可能性がある

アクセスごとに処理が走るため、利用するホスティングサービスによってはコストが増える場合があります。

### 4. キャッシュ設計が必要になる

全ユーザーに同じ内容を返すページならキャッシュしやすいです。

一方、ユーザーごとに内容が違うページはキャッシュしにくくなります。

## 21. SSRを使うかどうかの判断基準

SSRを使うか迷った場合は、以下の基準で判断します。

| 判断ポイント          | 推奨               |
| --------------- | ---------------- |
| 内容が固定           | SSG / 静的配信       |
| 記事や固定ページ中心      | SSG / 静的配信       |
| SEOが重要で、内容が動的   | SSR              |
| ログイン状態で表示を変える   | SSR              |
| ユーザーごとに内容が違う    | SSR              |
| アクセス時点の最新データが必要 | SSR              |
| 管理画面などSEOが不要    | CSRでも可           |
| APIやデータを守りたい    | サーバー側/API側の認可が必須 |

## 22. まとめ

SSRとは、**アクセス時にサーバー側でHTMLを生成して返す方式**です。

SSRの主なメリットは以下です。

```txt
SEOに強い
初回表示が速く感じやすい
ログイン状態による表示分岐がしやすい
最新データを反映しやすい
APIキーやサーバー側処理を隠しやすい
```

特にログインが関係するページでは、SSRを使うことで、サーバー側でログイン状態を確認してからHTMLを返せます。

ただし、ログイン画面へ遷移させるだけならCSRでも可能です。

SSRが有利なのは、以下のような点です。

```txt
画面を返す前に判定できる
チラつきが起きにくい
不要なAPI通信を減らせる
最初から正しい画面を返せる
```

一方で、固定ページや記事ページのように内容があまり変わらないページでは、SSRを使わなくても問題ないことが多いです。

基本的には、以下の考え方で判断するとよいです。

```txt
固定ページ・記事ページ
→ SSG / 静的配信

ログイン状態で表示が変わるページ
→ SSRを検討

ユーザーごとに内容が違うページ
→ SSRを検討

SEOが不要な管理画面
→ CSRでも可
```

SSRはすべてのページに必要なものではなく、**動的な表示・ログイン判定・SEO・最新データが必要なページで使う選択肢**として理解するとよいです。

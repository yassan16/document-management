# tasks.md テンプレート

## 概要

このファイルは、SDD（仕様書駆動開発）のタスク分割フェーズで使用する `tasks.md` の **実用テンプレート** です。プロジェクトの `specs/tasks.md` としてコピーし、各セクションを記入してください。

書き方の詳細な解説やアンチパターンについては、[タスク分割フェーズガイド（03-tasks/guide.md）](./guide.md) を参照してください。

## 使い方

### 1. 規模を選ぶ

| 規模 | 対象 | 特徴 |
|---|---|---|
| **Small** | 個人プロジェクト・1人開発 | チェックリスト形式で素早く着手 |
| **Medium（推奨）** | 2〜5人のチーム開発 | バランスの取れた標準テンプレート |
| **Large** | 5人以上・複数チーム横断 | ポイント見積・担当ロール・リスク管理付き |

### 2. テンプレートをコピーする

該当する規模のテンプレートセクションをコピーし、プロジェクトの `specs/tasks.md` に貼り付けてください。

### 3. 各セクションを記入する

`<!-- ... -->` のコメントに従って記入してください。記入が終わったらコメントは削除して構いません。

### 4. Gate3 チェックリストで確認する

テンプレート末尾のチェックリストで品質を確認し、全項目を満たしたら実装フェーズに進みます。

## テンプレート（Small - 個人プロジェクト向け）

個人プロジェクト向けの軽量テンプレートです。チェックリスト形式で最低限のタスクを記述します。

````markdown
# [プロジェクト名] — tasks.md

## タスク一覧

<!-- 「タスク名: 完了条件」の形式で箇条書きにする -->
<!-- 依存がある場合は「(→ T-XX 完了後)」と添える -->

- [ ] T-01: <!-- タスク名 -->
  - <!-- 完了条件: 〜できること -->
- [ ] T-02: <!-- タスク名 -->（→ T-01 完了後）
  - <!-- 完了条件: 〜できること -->
- [ ] T-03: <!-- タスク名 -->（→ T-02 完了後）
  - <!-- 完了条件: 〜できること -->
- [ ] T-04: <!-- タスク名 -->（→ T-01 完了後）
  - <!-- 完了条件: 〜できること -->

## Gate3 セルフチェック

- [ ] 全タスクに完了条件が書かれている
- [ ] 各タスクが1日以内で完了できる粒度になっている
- [ ] 依存関係が把握できている
- [ ] requirements.md の全機能要件が最低1つのタスクに対応している
- [ ] 読み返して「次に何をすべきか」が自分で判断できる
````

## 参考: 記入のコツ

テンプレートを記入する際に役立つポイントをまとめます。詳細は [03-tasks/guide.md](./guide.md) を参照してください。

### タスクIDの命名規則

| ルール | 説明 | 例 |
|---|---|---|
| 連番 | T-01 から始め、1ずつ増やす | T-01, T-02, T-03 |
| ゼロ埋め | 2桁でゼロ埋めする（10件以上の場合） | T-01〜T-10 |
| フェーズ分け | フェーズが複数ある場合はプレフィックスを使う | T1-01（フェーズ1）, T2-01（フェーズ2） |

### 依存関係の書き方

| パターン | 書き方 | 意味 |
|---|---|---|
| 単一依存 | T-01 | T-01が完了してから着手できる |
| 複数依存 | T-01, T-02 | T-01とT-02の両方が完了してから着手できる |
| 依存なし | — | いつでも着手できる |

依存関係は「技術的な依存（スキーマがないとRepositoryが作れない）」と「ロジック依存（ServiceにはRepositoryのインターフェースが必要）」の2種類を意識してください。

### DoDの粒度

| レベル | 例 | 評価 |
|---|---|---|
| ❌ 曖昧 | 「Todo機能が動く」 | 「動く」の定義が不明 |
| ❌ 手順指定 | 「`useState` で状態管理する」 | 実装方法の指定はDoD不要 |
| ✅ 適切 | 「タイトルを入力して送信すると一覧にTodoが追加される」 | テスト可能な具体的動作 |
| ✅ 適切 | 「タイトルが空の場合、エラーメッセージが表示される」 | エラーケースも含む |

### 受け入れテストの書き方

受け入れテストは「**具体的な操作 → 期待される結果**」の形式で書きます。

```markdown
<!-- 悪い例: 抽象的すぎる -->
- ケース1: 正常動作を確認する

<!-- 良い例: 具体的な操作と結果 -->
- ケース1: フォームに「Buy milk」を入力して送信 → 一覧に「Buy milk」が追加される
- ケース2: 空のフォームを送信 → 「タイトルは必須です」がフォームの下に表示される
```

### よくあるアンチパターン

| パターン | 問題 | 対策 |
|---|---|---|
| 粒度が大きすぎる | AIが何から始めるかを自分で判断する | 1タスク1成果物に分割する |
| 完了条件が曖昧 | AIが「動けばOK」と判断する | 「〜できる/表示される」で具体化する |
| 依存が未定義 | 順序の取り違えで手戻りが発生する | 全タスクに依存を明示する |
| 要件との断絶 | AIが仕様書を参照しない | `FR-XX` のリファレンスを必ず付ける |
| 実装手順の混入 | 設計書のアーキテクチャを無視した実装になる | DoD（何を）と実装ガイド（方向性）を分ける |

## 1. 基本方針

本書では、要件定義とその前段の要求形成を、実務上分かりやすいように次の3段階で整理する。

| 段階 | 目的                       |
| -- | ------------------------ |
| 初期 | ビジネス上の目的と利害関係者のニーズを明確にする |
| 中期 | システムに必要な要件を分析・構造化する      |
| 後期 | 要件を詳細化・検証し、関係者で合意する      |

「初期・中期・後期」は、規格上の正式なフェーズ名ではない。

最新のSLCPにおける、次のプロセスを実務向けにまとめた区分である。

```text
ビジネス又はミッション分析
    ↓
利害関係者ニーズ及び要件定義
    ↓
システム要件定義
    ↓
アーキテクチャ・設計定義
```

各段階は一方向ではなく、検討結果に応じて反復する。

```text
初期 ⇄ 中期 ⇄ 後期
```

## 2. 初期：目的とニーズの明確化

### 目的

次の点を明確にする。

```text
なぜ変更が必要なのか
誰にどのような価値を提供するのか
何を対象とするのか
```

### 決めること

* 背景、課題、原因
* 現状業務と現行システム
* 実現したい将来像
* 目的と成功指標
* 利用者、運用者、決裁者などの関係者
* 対象業務、対象組織、対象外範囲
* システム境界の概要
* 期限、予算、法令、既存環境などの制約
* 主要なリスク
* 要求の優先順位
* 重要な非機能要求
* 外部連携、データ移行、運用の有無

### 主な成果物

* プロジェクト目的
* 現状と将来業務の概要
* スコープ定義
* ステークホルダー一覧
* ビジネス要求一覧
* 成功指標
* 前提・制約・リスク一覧
* システムコンテキスト図

---

## 3. 中期：システム要件の分析・構造化

### 目的

次の点を明確にする。

```text
誰が、どの業務を、どの条件で行うのか
システムは何を提供する必要があるのか
```

### 決めること

* システム導入後の業務フロー
* 利用者の役割
* ユースケースと機能一覧
* 業務用語とエンティティ
* エンティティ間の関係
* エンティティのライフサイクル
* 状態遷移
* 業務ルール
* 操作権限とデータ参照範囲
* 必要なデータ
* 外部システムとの責任分界
* 主要な正常系と例外系
* 非機能要件の利用シナリオ
* データ移行方針
* 運用方針
* 業務レベルの受入条件
* 要件間の優先順位と依存関係

### 主な成果物

* To-Be業務フロー
* ユースケース・機能一覧
* 概念データモデル
* 状態遷移図
* 業務ルール一覧
* 権限表
* 外部連携一覧
* 非機能要求一覧
* 受入シナリオ

## 4. 後期：要件の詳細化・検証・合意

### 目的

要件を次の状態まで具体化する。

```text
曖昧さがない
必要性が説明できる
実現可能である
テストによって確認できる
関係者が同じ意味で理解している
```

### 決めること

* 機能の実行条件と処理結果
* 入力情報と出力情報
* 項目の必須・任意、値域、文字数、重複条件
* 状態遷移の実行者、条件、結果
* エラー、例外、同時更新時の挙動
* 外部連携のデータ、タイミング、失敗時の扱い
* 性能、可用性、セキュリティなどの測定条件
* データ移行の対象、変換、照合方法
* 運用、監視、障害対応
* ログ、監査、データ保持期間
* テスト可能な受入条件
* 要件間の整合性
* 要件と設計・テストのトレーサビリティ
* 未決事項、対象外事項、将来対応
* 関係者による合意・承認

### 検証と妥当性確認

要件の確認は、次の2つの観点で行う。

| 観点    | 確認内容                |
| ----- | ------------------- |
| 検証    | 要件が明確、一貫、実現可能、検証可能か |
| 妥当性確認 | 利害関係者のニーズや目的を満たす要件か |

### 主な成果物

* 詳細要件一覧
* 項目定義
* 状態遷移条件表
* エラー・例外一覧
* 非機能要件
* 外部連携要件
* データ移行要件
* 運用要件
* 受入基準
* 要求トレーサビリティ表
* 未決事項一覧

## 5. 全段階を通して行うこと

次の項目は、後期になって初めて検討するのではなく、初期から段階的に具体化する。

| 項目       | 初期            | 中期          | 後期              |
| -------- | ------------- | ----------- | --------------- |
| スコープ     | 大枠を決める        | 業務・機能単位に分ける | 境界を確定する         |
| 機能要件     | 大分類を把握する      | 洗い出して構造化する  | 詳細条件を確定する       |
| ライフサイクル  | 主要状態を仮置きする    | 状態と遷移を定義する  | 遷移条件・権限・結果を確定する |
| 非機能要件    | 重要な品質要求を把握する  | 利用状況を整理する   | 測定可能な条件にする      |
| 外部連携     | 連携先の有無を確認する   | 責任分界を決める    | 方式と失敗時の処理を確定する  |
| データ移行    | 必要性と規模を確認する   | 対象と方針を決める   | 変換・照合条件を確定する    |
| 運用       | 運用主体と時間帯を確認する | 運用フローを決める   | 監視・障害対応を確定する    |
| 受入条件     | 成功指標を決める      | 業務シナリオを作る   | テスト可能な条件にする     |
| リスク      | 主要リスクを識別する    | 対応方針を決める    | 対応条件と担当を確定する    |
| トレーサビリティ | 要求の出所を記録する    | 要求間を関連づける   | 設計・テストとの対応を確認する |
| 変更管理     | 変更手続きを決める     | 影響範囲を評価する   | 合意・承認して反映する     |

## 6. 要件と設計の境界

要件定義では、主に「なぜ必要か」「何が必要か」を決める。

```text
販売停止中の商品は購入できないこと
商品名と商品コードで検索できること
権限のない利用者は販売開始できないこと
```

設計では、主に「どのように実現するか」を決める。

```text
販売停止状態を赤色のラベルで表示する
検索欄を画面上部に配置する
一覧を20件単位でページングする
```

ただし、業務、法令、アクセシビリティなどにより実現方法自体が必須である場合は、要件として定義する。

---

## 7. まとめ

```text
初期
目的、価値、対象範囲、関係者のニーズを明確にする
    ↓
中期
業務、機能、データ、状態、ルール、権限を構造化する
    ↓
後期
要件を実現・検証・合意可能な条件まで具体化する
```

非機能要件、外部連携、データ移行、運用、受入条件、リスク、トレーサビリティ、変更管理は、初期から継続的に検討する。
